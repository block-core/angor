Why the Original Feels Like Spaghetti

Three paradigms fighting in one codebase

The original mixes DDD aggregates, functional monads (Result/Maybe chaining), and ReactiveUI streams — often in the same method. Developers must hold three mental models simultaneously to trace a single user action.

// Real type from the original codebase:
Task<Result<Maybe<Result>>>
// Three nested monadic layers. Wrapping wrappers.

Interface ceremony with no polymorphism

Nearly every ViewModel has a paired IFooViewModel interface with exactly one implementation. None are ever swapped at runtime. This doubles the file count and forces 85 *Sample.cs design-time stubs that duplicate every ViewModel.

Factory classes for one-liners

// ProjectInvestCommandFactory.Create() — the entire body:
var command = EnhancedCommand.CreateWithResult(
    () => navigator.Go<IInvestViewModel>(), canExecute);
// A one-liner disguised as a factory + interface + DI registration.

Tracing an invest flow: 12+ files

Button → DetailsViewModel → Navigator → FooterViewModel.InvestFlow() (40-line god method) → PaymentSelectorViewModel → InvoiceViewModel → Dialog. Scattered across Flows/, Sections/, Factories/, and Core/.

Where Avalonia2 Wins

Architecture

  • One paradigm — ReactiveUI + simple methods
  • Flows are self-contained in single section folders
  • No interface ceremony or factory bloat
  • Direct SDK calls without indirection layers

User Experience

  • Mobile-first responsive layout (3 breakpoints)
  • Bottom tab bar with sub-tab state memory
  • View caching with onReuse callbacks
  • Material 3-style token theme system
  • Android AOT + trimming build pipeline

What Avalonia2 Must Fix Before Production

Engineering Gaps

  • double used for BTC amounts — will produce rounding errors
  • ViewModels with subscriptions are never disposed — memory leaks
  • 10 bare catch blocks silently swallow exceptions
  • Prototype data still wired (fake npubs, sample flags)

Why These Matter

  • Floating-point money is a known source of lost funds
  • Undisposed subscriptions cause ghost updates in long sessions
  • Silent catches make production bugs invisible
  • Prototype data left in shipping code confuses users

Concrete fixes

  • Replace double with integer sats
    20+ locations using double Balance, double.TryParse for amounts
  • Add CompositeDisposable to all subscribing ViewModels
    65 disposal refs vs the original's 274 — subscriptions are leaking
  • Replace bare catch blocks with logging
    10 silent swallowers across 6 files, including catch { } in production
  • Remove prototype/sample data
    SignatureStore fake npubs, PrototypeSettings.ShowPopulatedApp, Guid investor names

Verdict

The original AngorApp is well-intentioned architecture that became over-engineered — interface ceremony, wrapper types around wrapper types, factory classes for one-liners, and three conflicting paradigms that make it hard to reason about flows. The 85 sample files are particularly telling: they exist because the design-time data pattern requires interface implementations, creating a duplication tax on every feature.

Avalonia2 correctly rejected the abstraction overhead. It has real mobile-first functionality the original lacks and is directly wired to the SDK without indirection layers. Its gaps — disposal, error handling, floating-point money, leftover prototype data — are concrete and fixable. They're missing plumbing, not architectural debt.

The decision to rewrite rather than refactor the original is defensible.

Bottom line: Avalonia2 is the right direction. Ship it — after fixing the four items above.
