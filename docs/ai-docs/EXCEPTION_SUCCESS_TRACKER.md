# Exception And Success UX Tracker

Last crawl: 2026-06-24

Scope crawled:

- New Avalonia rewrite: `src/design/App`
- Primary Avalonia app: `src/avalonia/AngorApp`, `src/avalonia/AngorApp.Model`
- SDK and shared services: `src/sdk/Angor.Sdk`, `src/shared/Angor.Shared`
- Legacy Blazor app: `src/webapp/Angor.Client`

Purpose:

- Track every known place where the app can fail, throw, show a raw exception, silently swallow a problem, or succeed with user-facing feedback.
- Decide one by one which component should handle each case and what wording should be shown.
- Keep a layman explanation of the user journey that leads to each failure, so copy and UI choices are understandable without reading code.
- Work through each case as a question-and-answer scenario: what is the user trying to do, what went wrong or right, what should the app ask/tell them, and what can they do next?

Local preview hook:

- Debug-only recovery exception lab added behind `ANGOR_EXCEPTION_UX_LAB=1`.
- Run with `ANGOR_EXCEPTION_UX_LAB=1 dotnet run --project src/design/App.Desktop/App.Desktop.csproj`.
- Open `Funded` / Portfolio and click `Exception UX Lab`.
- The lab previews recovery question, error, and success states using canned data and does not call the SDK.

Recovery Q/A cases currently previewable:

| Case | User Question | UI Answer To Decide |
|---|---|---|
| Recover without penalty | Founder signatures are available. Is this a safe release action rather than a scary recovery action? | Release-confirmation modal, clear amount/destination/fee, error banner only if transaction build/publish fails. |
| End-of-project claim | Project has ended. Is this a normal claim, not penalty recovery? | Claim wording, no penalty warning, explain indexer/output-not-found as a retryable sync issue. |
| Below-threshold direct recovery | Investment is below penalty threshold. Should recovery skip penalty warning? | Direct recovery question, no penalty-lock copy, wallet/change-address failures as actionable wallet retry. |
| Recover to penalty | User can recover now but funds will lock for the penalty period. What warning is sufficient? | Recovery warning modal with penalty duration, amount, fee, and explicit lock-period consequence. |
| Recover from penalty | Penalty period has ended. Should this use release wording? | Release wording, avoid repeating initial recovery warning, explain unconfirmed recovery transaction as prerequisite. |

Status values:

- `Open`: needs product/design/engineering decision.
- `Decided`: component and wording chosen.
- `Fixed`: code changed and verified.
- `Deferred`: accepted for later.

Component decision examples:

- Inline field validation
- Inline flow error banner
- Modal alert
- Toast/notification
- Empty state
- Disabled action hint
- Log only
- Retryable blocking error
- Success modal
- Success toast

| Status | ID | Problem | Current Pattern | Affected Rows | Decision Needed |
|---|---|---|---|---|---|
| Open | X-RAW-EXCEPTION | Raw exception text is shown to users. | Many paths use `{ex.Message}`, raw SDK errors, raw relay/indexer HTTP bodies, or technical operation names. | Most `*-EXCEPTION`, SDK rows, webapp rows. | Decide user-safe generic copy plus optional technical details component. |

| Open | X-SILENT-FAILURE | Some failures are swallowed or log-only. | Services catch/log and return empty/partial success, or commands have no visible error handling. | `AV-FIND-001`, `AV-FUNDED-001`, `AV-SETTINGS-001`, `SDK-FOUNDER-006`, `WEB-MESSAGE-001`, `WEB-SIGN-001`, `WEB-UNFUNDED-001`. | Decide what must be visible versus log-only. |
| Open | X-DUPLICATE-SUCCESS | Some flows show duplicate success states. | Success toast plus success modal, or success in `finally`. | `AV-DEPLOY-003`, `AV-MANAGE-002`, `WEB-WALLET-002`. | Decide one success component per flow. |
| Open | X-WRONG-CONTEXT | Some generic components use misleading copy. | Transaction previewer says `Failed to submit investment offer` for deploy/recovery/claim. | `AV-DEPLOY-002`, `AV-FUNDED-005`. | Parameterize transaction previewer failure copy by operation. |

## Suggested Review Order

1. Decide cross-cutting error component rules in `X-*` rows.
2. Fix new app payment/deploy rows first: `DESIGN-PAY-*`, `DESIGN-DEPLOY-*`.
3. Fix primary app transaction previewer wording: `AV-DEPLOY-002`, `AV-FUNDED-005`, `X-WRONG-CONTEXT`.
4. Normalize SDK messages that bubble most often: `SDK-WALLET-*`, `SDK-TX-001`, `SDK-INVEST-*`, `SDK-RECOVERY-*`.
5. Decide whether legacy webapp gets full cleanup or only safety fixes: `WEB-*`.
